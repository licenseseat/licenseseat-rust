# Release 0.5.3

## Summary

`0.5.3` is the hardening follow-up to `0.5.2`.

`0.5.2` introduced trusted offline metadata snapshots, but production integration showed that
the SDK could still downgrade a valid paid license into a metadata-empty offline license when two
conditions happened together:

1. the cached machine file verified successfully but omitted its embedded license object
2. the dedicated `license_snapshot` cache file was missing

`0.5.3` removes that remaining single-point dependency.

The SDK now persists trusted rich license metadata on the cached license record itself and uses
that metadata as a durable fallback during offline restore.

## What Changed

### Trusted metadata is now persisted in two places

Successful online license-bearing responses now preserve trusted rich metadata through:

- the dedicated `license_snapshot` cache file
- the cached `license.json` record itself

This means the SDK no longer depends on a single sidecar file to preserve tier metadata offline.

### Offline restore now has an explicit recovery order

For stripped machine-file payloads, offline restore now resolves metadata in this order:

1. embedded machine-file license
2. snapshot-file metadata
3. trusted cached license metadata on `license.json`
4. blank fallback only as a true last resort

That is the important behavioral guarantee for tiered desktop apps.

### Snapshot files now self-heal

If offline restore finds trusted cached license metadata but the dedicated snapshot file is gone,
the SDK recreates the snapshot file before continuing.

This does not invent metadata that was never seen online. It only makes already-trusted metadata
durable again.

### Tauri admin/debug output is clearer

The Tauri admin snapshot now exposes:

- `trustedLicense`
- `trustedLicenseSource`
- `cachePaths.licenseSnapshotPath`

This separates:

- the best trusted metadata currently available to the SDK
- from the path of the dedicated snapshot file on disk

## Why This Release Was Needed

The real production failure mode was narrower than a general offline-validation failure:

- the machine file was valid
- offline restore succeeded
- the license still came back `offline_valid`
- but `plan_key`, entitlements, and product metadata were blank

Downstream apps then interpreted that state as a lower tier.

The fix needed to live in the SDK because the public app/runtime state is derived from the SDK's
resolved validation result.

## Regression Coverage

This release adds direct coverage for the production-shaped edge case:

1. activate or validate or heartbeat online
2. fetch a stripped machine file
3. delete the dedicated `license_snapshot` file
4. restart the SDK
5. restore offline
6. assert that plan, entitlement, and product metadata still match the last trusted online state

Covered paths:

- activation-only metadata recovery
- validation-refreshed metadata recovery
- heartbeat-refreshed metadata recovery
- Tauri admin snapshot exposure for trusted metadata
