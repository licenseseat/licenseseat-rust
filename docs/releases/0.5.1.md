# Release 0.5.1

## Summary

`0.5.1` is the parity and production-hardening release for the Rust SDK and the Tauri plugin.

The main goals of this release are:

1. Bring the Rust core SDK to feature parity with the C++ reference implementation.
2. Bring the Tauri plugin surface much closer to parity with the Rust core crate instead of leaving it as a materially smaller wrapper.
3. Align the Rust request/response shape with the current LicenseSeat API where parity gaps existed.
4. Harden the defaults so mixed-SDK estates identify devices and recover offline state consistently.

This is not a cosmetic release. It changes the default fingerprinting strategy, expands the runtime and offline APIs, widens the Tauri command surface, and adds significant new regression coverage.

## Release Scope

This release includes work in four major areas:

1. Core Rust SDK parity with the C++ reference implementation.
2. Offline-validation and restore-flow hardening.
3. Tauri plugin parity and frontend contract cleanup.
4. Documentation and test coverage upgrades for release confidence.

## High-Level Outcomes

### 1. Core SDK parity

The Rust SDK now exposes the major capabilities that existed in the C++ reference but were previously missing or incomplete on the Rust side:

- C++-style default fingerprint generation and richer fingerprint components.
- Explicit stateless `validate_key`, `deactivate_key`, and `heartbeat_key` flows.
- Separate runtime status helpers and more granular background-task lifecycle control.
- Release APIs and download-token APIs.
- Machine-file checkout, verification, restore, and signing-key fetch/cache flows.
- Better event coverage for auth failures, auto-validation failures, offline flows, and SDK/runtime errors.

### 2. Offline model hardening

Offline validation moved from a legacy-token-centric implementation to a machine-file-first model with safer fallback behavior:

- machine-file checkout and verification are first-class flows
- signing keys can be fetched and cached automatically
- restore behavior can recover cached sessions more reliably
- legacy offline tokens remain available for manual and fallback workflows
- fallback behavior now preserves the intended order instead of being short-circuited by intermediate setup failures

### 3. Tauri parity and frontend contracts

The Tauri layer is no longer missing the major manual offline and release-management APIs from the Rust core:

- restore, health, client status, fingerprint access, and online-state APIs are exposed
- release listing, latest release, and download-token APIs are exposed
- manual legacy offline-token generation and verification are exposed
- manual machine-file verification is exposed
- Tauri event payload forwarding is documented and tested
- public status strings now match the stable snake_case core contract

### 4. Production readiness

This release emphasizes production-safe behavior, not just feature count:

- the same machine now identifies much more consistently across Rust and C++
- support/background tasks no longer duplicate after stop/start timing windows
- public docs now reflect the actual event and status contracts
- plugin-side command conversion behavior has direct tests rather than only compile coverage

## Behavior and Compatibility Notes

### Fingerprint behavior changed

The Rust SDK default fingerprint is intentionally different in this release.

Previous Rust behavior:

- generated a `rust_<...>`-style identifier
- relied on a narrower set of device signals
- did not align with the C++ reference output shape

New Rust behavior:

- uses a prioritized set of platform signals that mirrors the C++ reference
- hashes the selected raw signal with SHA-256
- returns the first 32 lowercase hex characters
- emits structured fingerprint components that can be reused for diagnostics and machine-file requests

Impact:

- mixed estates using both Rust and C++ now have much better default device identity compatibility
- apps relying on the old Rust-only fingerprint shape should set an explicit custom fingerprint if they need to preserve that prior identity scheme

### Tauri status strings changed

The Tauri plugin now exposes:

- `offline_valid`
- `offline_invalid`

instead of the previous camelCase values:

- `offlineValid`
- `offlineInvalid`

Impact:

- this is the correct long-term contract because it matches the core Rust SDK and the C++ reference semantics
- frontend consumers that were matching on the old camelCase values should update before release

### Plugin release sequencing matters

The Rust Tauri crate depends on `licenseseat = ^0.5.1`.

That means the release order should be:

1. publish `licenseseat` to crates.io
2. wait for the crates.io index to pick up `0.5.1`
3. publish `tauri-plugin-licenseseat`
4. publish `@licenseseat/tauri-plugin` to npm

The JavaScript package is ready to pack independently, but the Rust plugin crate cannot complete packaging/publishing until the core crate version exists on crates.io.

## Detailed Technical Changes

### Workspace and package metadata

- Workspace version advanced to `0.5.1`.
- Rust crate metadata and package docs were updated to describe the expanded offline/release surface.
- The Tauri npm package version was aligned to `0.5.1`.

### Core Rust SDK: device identity

The device-identity layer was substantially reworked.

Key changes:

- new `device.rs` module centralizes canonical fingerprint generation
- default fingerprint selection now mirrors the C++ strategy
- fingerprint component collection now captures richer platform metadata
- `device_id`, `fingerprint`, and `device_fingerprint` aliases are handled more consistently
- empty configured identifiers no longer suppress the generated fingerprint path

Primary practical result:

- a machine that activates under the C++ SDK is much more likely to resolve to the same identity when using the Rust SDK by default

### Core Rust SDK: runtime and lifecycle control

The runtime model now more closely matches the C++ reference while still staying idiomatic for Rust.

Key changes:

- explicit runtime status helpers were added
- background support tasks were separated more cleanly from validation/heartbeat timers
- generation counters were introduced to prevent stale support loops from surviving stop/start cycles
- lifecycle events for auth failures, auto-validation failures, and runtime errors were added

Primary practical result:

- stop/start behavior is safer in long-lived desktop apps
- event-driven integrations can react to more of the runtime lifecycle without guessing

### Core Rust SDK: offline and restore flows

This release heavily expands and hardens the offline model.

Key changes:

- machine-file checkout became a first-class API
- machine-file verification became a first-class local API
- signing-key fetch/cache flows were added
- restore/session recovery APIs were added
- legacy offline-token generation and verification remain available
- fallback order was corrected so machine-file failures do not incorrectly suppress legacy-token fallback when that fallback is enabled
- machine-file request and payload handling were aligned more closely to the current API shape

Primary practical result:

- offline operation is now significantly more complete for production apps, especially when machine files are the primary distribution mechanism

### Core Rust SDK: release and download APIs

The Rust SDK now includes the release-management surface expected from the C++ reference:

- latest release lookup
- paginated release listing
- release download-token generation

Primary practical result:

- desktop installers, auto-updaters, and entitlement-gated download flows can now be implemented from the Rust SDK directly

### Core Rust SDK: models and public API surface

The public type surface was widened so parity features can be consumed directly and, where necessary, bridged into the Tauri plugin cleanly.

Key changes:

- more response and model types are re-exported from the crate root
- release envelopes and offline payload types are modeled more explicitly
- status and event types now cover more runtime states

Primary practical result:

- downstream apps and the Tauri plugin do not need to depend on internal/private model modules to use advanced flows

### Tauri plugin: command surface

The Tauri plugin gained a major surface-area expansion.

Newly exposed or significantly improved command families:

- explicit validation/deactivation/heartbeat helpers
- client status and online-state helpers
- fingerprint access
- restore and health
- latest release, list releases, and download-token generation
- machine-file checkout
- signing-key fetch
- legacy offline-token generation and verification
- machine-file verification

Primary practical result:

- the plugin is now a much more faithful frontend-facing companion to the Rust core SDK instead of a narrow wrapper around only the earliest lifecycle operations

### Tauri plugin: JavaScript bindings

The guest JS bindings were expanded to match the new command surface and to expose richer typed payloads for:

- releases
- download tokens
- machine files
- machine-file verification
- offline tokens
- restore results

Primary practical result:

- TypeScript consumers can integrate the expanded API surface without hand-rolling their own payload types

### Tauri plugin: event bridge and docs

The plugin already forwarded backend events to Tauri’s event system, but the docs were out of sync with the runtime payloads.

This release fixes that mismatch:

- `validation-failed` is documented as a structured `ValidationResult` payload
- function listings now include the manual offline helper APIs
- status values in the docs reflect the stable snake_case contract

### Test and verification upgrades

This release adds regression coverage for:

- C++-style fingerprint priority and shape
- support-task restart races
- Tauri plugin payload conversions for offline tokens and machine files
- structured Tauri event payload forwarding for validation failures
- expanded release/list/download-token APIs
- machine-file request/response alignment

Primary practical result:

- the new parity surface is backed by targeted tests instead of only broad integration assumptions

## PR Framing

If this release is shipped through a GitHub PR, the PR should frame the change as:

- a parity release against the C++ reference implementation
- a production-hardening release for fingerprint identity, background lifecycle, and offline recovery
- a Tauri parity release that removes the largest remaining frontend-surface gaps

That framing matters because the diff is large, but it is cohesive: it is all in service of making the Rust repo a complete, production-ready peer to the C++ SDK rather than a partially aligned implementation.
