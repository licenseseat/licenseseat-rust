# Release 0.5.2

## Summary

`0.5.2` is a focused patch release for the Rust SDK and the Tauri plugin.

The release fixes an offline restore regression discovered during a real app integration:
when a verified machine file omitted its embedded license object, the SDK could restore an
offline-valid license while losing the plan, entitlement, and product metadata that downstream
apps rely on for tiered UX and feature gating.

The release fixes that issue without changing the SDK's activation semantics.

## What Changed

### Trusted license snapshots

The SDK now maintains a trusted cached `LicenseResponse` snapshot from successful online
license-bearing responses:

- activation
- validation
- heartbeat

That snapshot is stored separately from the cached `License.validation` state.

### Offline machine-file enrichment

Offline machine-file validation now hydrates from the trusted cached snapshot only when:

- the machine file verifies successfully
- the verified payload omitted its embedded license object
- the cached snapshot matches the same license key

If the machine file includes its own embedded license object, that embedded license remains the
source of truth.

### Semantics preserved

The release deliberately does not treat activation as validation.

After activation:

- the cached license still has `validation: None`
- `status()` still reports `Pending`
- offline restore can still recover richer plan and entitlement metadata because it uses the
  separate trusted snapshot, not a synthetic validation result

## Why This Shape

The earlier fix attempt solved the UI symptom by seeding validation state during activation.
That made offline restore look correct in apps, but it changed public SDK semantics and broke the
existing activate-then-pending contract.

`0.5.2` fixes the actual problem instead:

- keep trusted metadata separately
- use it only for stripped offline machine-file results
- keep activation and validation behavior unchanged

## Regression Coverage

The SDK integration suite now explicitly covers:

- pending status after activation
- offline restore from an activation-only snapshot
- offline restore from a validation-refreshed snapshot
- offline restore from a heartbeat-refreshed snapshot
- precedence of embedded machine-file license data over cached snapshots
